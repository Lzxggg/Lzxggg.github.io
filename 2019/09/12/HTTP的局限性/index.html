HTTP的局限性

HTTP是无状态、无连接、单向的应用层协议,采用请求、响应模型。这种设计，导致如果服务器状态联系发生变化需要告知客户端、非常麻烦。常用的解决方案有轮询和长轮询。

  • 轮询：每隔一段时间请求服务器一次，看看有没有新消息，服务器立马返回响应消息（没有新消息，或者新消息）

  • 长轮询：在轮询的基础上，服务器收到请求后阻塞，等到有新消息时才返回。（也称为 Comet ，一种模拟服务器端推送的技术）

    轮询特别考验服务器的处理性能（速度），而长轮询考验服务器的能同时处理的请求数（空间).两种方式都会频繁发送请求，建立连接，再断开，如此循环，消耗大量的服务器资源

在HTTP 1.0 中，每个HTTP请求和相应都需要建立一次TCP连接。在HTTP1.1中引入了keepalive，可以在一次TCP中发送多个请求，收到多次响应，提升了传输效率。但是这还不够为了解决这一问题，有了WebSockt

WebSockt

WebSockt是一种基于TCP链接，利用HTTP协议进行握手的连接方式。

  • 可以发送文本或者二进制数据
  • 没有同源限制
  • 协议标识符时ws，加密则为wss

WebSockt 握手过程

客户端发送一个HTTP请求，请求头相比以往多了如下几个字段：

  • Upgrade：websocket
  • Connection：Upgrade

这两个字段的意思时，告诉服务器要把链接升级为WebSocket链接。

Sec-WebSokectVersion：这个字段规定使用的版本

服务器返回的响应状态是101-Switching Protocols，响应头也包含上述Upgrade那部分，此外还有：

  • Sec-WebSokect-Accept: 用于完成key的验证
  • Sec-WebSokect-Protocol ：最后确定的协议（在请求的协议中选一个）

使用

继承自EventTarget，因此可以使用那三个方法。

  • 新建一个WebSocket实例，传入url和协议（单个string或者数组）
  • 为onopen、onmessage、onclose、onerror绑定handle
  • send（） 发送数据
  • close（）关闭链接

WebSocket的readyState

  • 0-CONNECTING
  • 1-OPEN
  • 2-CLOSING
  • 3-CLOSED
