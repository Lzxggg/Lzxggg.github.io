浏览器的结构

User Interface : 用户界面 除了请求得到的浏览器显示的主界面部分都是用户界面 包括导航栏 前进后退按钮 等

Browser engine ：浏览器引擎 用于 用户界面和渲染引擎之间的交互

Rendering engine：渲染引擎 用于构建dom树 cssom树

netWorking ：网络接口 比如 Http 协议等 在不同的浏览器都有底层实现

UI Backend ：用户界面后端

Data Persistence ： 也就是持久层 比如 Cookie H5 新增的一个完整的在浏览器的数据库 等

JS Interpreter ：JS解释器 用于解析和执行JS代码 Chrome 的解释器是V8

关于 Chrome 他为每一个标签页都建立了一个单独的进程

优点：

  • 稳定性：每当有死循环的JS或者其他原因使得崩溃 只有那一个标签页 其他标签页正常运行
  • 性能：使得 操作系统 和 硬件 更好的使得进程调度 提高浏览器的运行速度
  • 安全性：进程是一个独立的沙箱，减少意外操作导致损失的几率

缺点：

  • 开销太大

浏览器内核

内核 浏览器 备注
Trident IE 众多国产多核浏览器所谓“兼容模式”所使用的内核
Webkit Sarifa 早期Chrome 也用
Blink Chrome Opera Google和Opera合作的内核，用于Chrome和Opera最新系列浏览器
Presto Opera原来的内核 已经废弃，渲染速度达到了极致，但是放弃了兼容性

注：大部分国产浏览器 都是双核 一般是兼容模式 用 Trident 然后高速模式用 Webkit

渲染过程

渲染过程：

  • 解析html 构建dom树

  • 解析css 构建cssom树

  • 合并dom树和 cssom树

  • 执行js

  • 布局

  • 绘制

渲染阻塞

从上到下解析 Html

首先渲染dom树 在渲染dom树的时候 如果遇到 script 标签 则会停止dom树的渲染 要先解释完这个js文件

如果这个js文件里面还有操作cssom的话 就要构建cssom树 直到渲染完cssom树之后才会继续渲染dom树

原因：js可能会改动dom树和cssom树，继续解析可能会造成浪费。所以浏览器等js执行完毕，把其对dom和cssom的变动追加到已经解析的树上，再继续解析后面的

解决方法 ：

  • css优先 js置后 js一般放在页面底部

  • 给script增加defer或async属性

    延时脚本defer属性：解析DOM树和加载该script会并行进行，但是script的执行会等到所有元素解析完成后，DOMContentLoaded事件触发前执行，如果有多个脚本，他们的执行顺序仍然是加载顺序

    异步脚本async属性：解析DOM树和加载与执行该script会并行进行，加载完就立即执行。如果有多个脚本，他们的执行顺序不一定，谁先加载完谁执行。（一般很少用）
